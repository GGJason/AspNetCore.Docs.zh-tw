## <a name="adding-claims-checks"></a><span data-ttu-id="00768-113">新增宣告檢查</span><span class="sxs-lookup"><span data-stu-id="00768-113">Adding claims checks</span></span>

<span data-ttu-id="00768-114">以宣告為基礎的授權檢查是宣告式的-開發人員會將其內嵌在其程式碼內、針對控制器中的控制器或動作，指定目前使用者必須擁有的宣告，並選擇性地將宣告必須保留以存取要求的資源的值。</span><span class="sxs-lookup"><span data-stu-id="00768-114">Claim based authorization checks are declarative - the developer embeds them within their code, against a controller or an action within a controller, specifying claims which the current user must possess, and optionally the value the claim must hold to access the requested resource.</span></span> <span data-ttu-id="00768-115">宣告需求是以原則為基礎，開發人員必須建立並註冊表示宣告需求的原則。</span><span class="sxs-lookup"><span data-stu-id="00768-115">Claims requirements are policy based, the developer must build and register a policy expressing the claims requirements.</span></span>

<span data-ttu-id="00768-116">最簡單的宣告原則類型會尋找宣告是否存在，且不會檢查值。</span><span class="sxs-lookup"><span data-stu-id="00768-116">The simplest type of claim policy looks for the presence of a claim and doesn't check the value.</span></span>

<span data-ttu-id="00768-117">首先，您必須建立並註冊原則。</span><span class="sxs-lookup"><span data-stu-id="00768-117">First you need to build and register the policy.</span></span> <span data-ttu-id="00768-118">這會在授權服務設定過程中發生，這通常會在您的 `ConfigureServices()` *Startup.cs*檔案中納入。</span><span class="sxs-lookup"><span data-stu-id="00768-118">This takes place as part of the Authorization service configuration, which normally takes part in `ConfigureServices()` in your *Startup.cs* file.</span></span>

::: moniker range=">= aspnetcore-3.0"

```csharp
public void ConfigureServices(IServiceCollection services)
{
    services.AddControllersWithViews();
    services.AddRazorPages();

    services.AddAuthorization(options =>
    {
        options.AddPolicy("EmployeeOnly", policy => policy.RequireClaim("EmployeeNumber"));
    });
}
```

::: moniker-end

::: moniker range="< aspnetcore-3.0"

<span data-ttu-id="00768-121">`AuthorizeAttribute`屬性可以套用至整個控制器，在此實例中，只有符合原則的識別才允許存取控制器上的任何動作。</span><span class="sxs-lookup"><span data-stu-id="00768-121">The `AuthorizeAttribute` attribute can be applied to an entire controller, in this instance only identities matching the policy will be allowed access to any Action on the controller.</span></span>

## <a name="multiple-policy-evaluation"></a><span data-ttu-id="00768-129">多重原則評估</span><span class="sxs-lookup"><span data-stu-id="00768-129">Multiple Policy Evaluation</span></span>

<span data-ttu-id="00768-130">如果您將多個原則套用至控制器或動作，則在授與存取權之前，所有原則都必須通過。</span><span class="sxs-lookup"><span data-stu-id="00768-130">If you apply multiple policies to a controller or action, then all policies must pass before access is granted.</span></span> <span data-ttu-id="00768-131">例如：</span><span class="sxs-lookup"><span data-stu-id="00768-131">For example:</span></span>

```csharp

    }
```

<span data-ttu-id="00768-132">在上述範例中，任何符合原則的身分識別 `EmployeeOnly` 都可以存取 `Payslip` 動作，因為該原則會在控制器上強制執行。</span><span class="sxs-lookup"><span data-stu-id="00768-132">In the above example any identity which fulfills the `EmployeeOnly` policy can access the `Payslip` action as that policy is enforced on the controller.</span></span> <span data-ttu-id="00768-133">不過，為了呼叫動作，身分 `UpdateSalary` 識別必須*同時*滿足 `EmployeeOnly` 原則和 `HumanResources` 原則。</span><span class="sxs-lookup"><span data-stu-id="00768-133">However in order to call the `UpdateSalary` action the identity must fulfill *both* the `EmployeeOnly` policy and the `HumanResources` policy.</span></span>
