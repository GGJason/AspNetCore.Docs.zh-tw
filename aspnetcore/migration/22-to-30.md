### <a name="framework-reference"></a>架構參考

如以上所列的其中一個封裝所提供的 ASP.NET Core 功能，可做為`Microsoft.AspNetCore.App`共用架構的一部分。 *共用架構*是一組安裝在電腦上的元件（ *.dll*檔案），其中包含執行時間元件和目標套件。 如需詳細資訊，請參閱[共用的架構](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/) \(英文\)。

* 以 SDK 為目標`Microsoft.NET.Sdk.Web`的專案會隱含`Microsoft.AspNetCore.App`地參考架構。

這些專案不需要其他參考：

```xml
<Project Sdk="Microsoft.NET.Sdk.Web">
  <PropertyGroup>
    <TargetFramework>netcoreapp3.0</TargetFramework>
  </PropertyGroup>
    ...
</Project>
```

* `Microsoft.NET.Sdk`以或`Microsoft.NET.Sdk.Razor` SDK 為目標的專案應該將明確`FrameworkReference`新增`Microsoft.AspNetCore.App`至：

```xml
<Project Sdk="Microsoft.NET.Sdk.Razor">
  <PropertyGroup>
    <TargetFramework>netcoreapp3.0</TargetFramework>
  </PropertyGroup>

  <ItemGroup>
    <FrameworkReference Include="Microsoft.AspNetCore.App" />
  </ItemGroup>
    ...
</Project>
```

#### <a name="framework-dependent-builds-using-docker"></a>使用 Docker 的 Framework 相依組建

使用相依于 ASP.NET Core[共用架構](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/)之套件的主控台應用程式架構相依組建可能會提供下列執行階段錯誤：

```cli
It was not possible to find any compatible framework version
The specified framework 'Microsoft.AspNetCore.App', version '3.0.0' was not found.
  - No frameworks were found.
```

`Microsoft.AspNetCore.App`是包含 ASP.NET Core 執行時間的共用架構，而且只會出現在[dotnet/Core/aspnet](https://hub.docker.com/_/microsoft-dotnet-core-aspnet/) docker 映射上。  3\.0 SDK 藉由不包含共用架構中可用的程式庫複本，減少使用 asp.net core 的 framework 相依組建大小。  這最多可能節省 ~ 18MB，但需要 ASP.NET Core 執行時間必須存在/安裝才能執行應用程式。

若要判斷應用程式是否具有 ASP.NET Core 共用架構上的相依性（直接或間接），請檢查您的應用程式組建/發行期間所產生的 *.runtimeconfig.json json*檔案。 下列 JSON 檔案顯示 ASP.NET Core 共用架構的相依性：

```JSON
{
  "runtimeOptions": {
    "tfm": "netcoreapp3.0",
    "framework": {
      "name": "Microsoft.AspNetCore.App",
      "version": "3.0.0"
    },
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

如果您的應用程式使用 docker，請使用包含 ASP.NET Core 3.0 的基底映射。 例如： `docker pull mcr.microsoft.com/dotnet/core/aspnet:3.0` 。

## <a name="kestrel"></a>Kestrel

### <a name="configuration"></a>組態

將 Kestrel 設定遷移至所提供`ConfigureWebHostDefaults`的 web 主機產生器（*Program.cs*）：

```csharp
public static IHostBuilder CreateHostBuilder(string[] args) =>
    Host.CreateDefaultBuilder(args)
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.ConfigureKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseStartup<Startup>();
        });
```

如果應用程式以手動方式`HostBuilder`建立主機，請在`ConfigureWebHostDefaults`的 web 主機 builder 上呼叫`UseKestrel` ：

```csharp
public static void Main(string[] args)
{
    var host = new HostBuilder()
        .UseContentRoot(Directory.GetCurrentDirectory())
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.UseKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseIISIntegration()
            .UseStartup<Startup>();
        })
        .Build();

    host.Run();
}
```

### <a name="connection-middleware-replaces-connection-adapters"></a>連線中介軟體會取代連接介面卡

已從 Kestrel<xref:Microsoft.AspNetCore.Server.Kestrel.Core.Adapter.Internal.IConnectionAdapter>移除連接介面卡（）。 以連線中介軟體取代連接介面卡。 連接中介軟體類似于 ASP.NET Core 管線中的 HTTP 中介軟體，但適用于較低層級的連接。 HTTPS 和連線記錄：

* 已從連接介面卡移至連線中介軟體。
* 這些擴充方法的作用如同舊版的 ASP.NET Core。 

如需詳細資訊，請參閱[Kestrel 文章的 listenoptions 來（通訊協定）一節中的 TlsFilterConnectionHandler 範例](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#listenoptionsprotocols)。

### <a name="transport-abstractions-moved-and-made-public"></a>已移動並設為公用的傳輸抽象概念

Kestrel 傳輸層已公開為中`Connections.Abstractions`的公用介面。 作為這些更新的一部分：

* `Microsoft.AspNetCore.Server.Kestrel.Transport.Abstractions`和相關聯的類型已移除。
* <xref:Microsoft.AspNetCore.Server.Kestrel.KestrelServerOptions.NoDelay>已從<xref:Microsoft.AspNetCore.Server.Kestrel.Core.ListenOptions>移至傳輸選項。
* <xref:Microsoft.AspNetCore.Server.Kestrel.Transport.Abstractions.Internal.SchedulingMode>已從中<xref:Microsoft.AspNetCore.Server.Kestrel.KestrelServerOptions>移除。

如需詳細資訊，請參閱下列 GitHub 資源：

* [用戶端/伺服器網路抽象概念（aspnet/AspNetCore #10308）](https://github.com/aspnet/AspNetCore/issues/10308)
* [在上層執行新的探源接聽程式抽象概念和重新平臺 Kestrel （aspnet/AspNetCore #10321）](https://github.com/aspnet/AspNetCore/pull/10321)

### <a name="kestrel-request-trailer-headers"></a>Kestrel 要求尾端標頭

針對以舊版 ASP.NET Core 為目標的應用程式：

* Kestrel 會將 HTTP/1.1 區塊尾端標頭新增至要求標頭集合。
* 結尾會在讀取要求主體之後提供。

這會導致一些關於標頭和結尾的多義性問題，因此結尾已移至3.0 中的新`RequestTrailerExtensions`集合（）。

HTTP/2 要求尾端為：

* 不適用於 ASP.NET Core 2.2。
* 以 3.0 `RequestTrailerExtensions`提供。

有新的要求擴充方法可存取這些結尾。 如同 HTTP/1.1，結尾會在讀取要求主體之後提供。

針對3.0 版本，可以使用下列`RequestTrailerExtensions`方法：

* `GetDeclaredTrailers`&ndash;取得要求`Trailer`標頭，其中列出主體之後預期的尾端。
* `SupportsTrailers`&ndash;指出要求是否支援接收尾端標頭。
* `CheckTrailersAvailable`&ndash;檢查要求是否支援尾端，以及是否可供讀取。 這種檢查並不會假設有尾端可供讀取。 即使這個方法傳回`true` ，也可能沒有尾端可讀取。
* `GetTrailer`&ndash;從回應中取得要求的尾端標頭。 請`SupportsTrailers`在呼叫`GetTrailer`之前檢查， <xref:System.NotSupportedException>如果要求不支援尾端標頭，則可能會發生。

如需詳細資訊，請參閱[將要求尾端放在個別的集合中（aspnet/AspNetCore #10410）](https://github.com/aspnet/AspNetCore/pull/10410)。

### <a name="allowsynchronousio-disabled"></a>AllowSynchronousIO 已停用

`AllowSynchronousIO`啟用或停用`HttpRequest.Body.Read`同步 IO api，例如、 `HttpResponse.Body.Write`和。 `Stream.Flush` 這些 Api 是執行緒耗盡的來源，導致應用程式損毀。 在3.0 中`AllowSynchronousIO` ，預設為停用。 如需詳細資訊，請參閱[Kestrel 文章中的同步 IO 一節](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#synchronous-io)。

除了啟用`AllowSynchronousIO` `ConfigureKestrel`的選項之外，同步 IO 也可以依個別要求覆寫，以做為暫時的緩和措施：

```csharp
var syncIOFeature = HttpContext.Features.Get<IHttpBodyControlFeature>();

if (syncIOFeature != null)
{
    syncIOFeature.AllowSynchronousIO = true;
}
```

如果您對於<xref:System.IO.TextWriter>在[Dispose](/dotnet/standard/garbage-collection/implementing-dispose)中呼叫同步 api 的程式或其他資料流程遇到問題，請改<xref:System.IO.Stream.DisposeAsync*>為呼叫新的 api。

如需詳細資訊，請參閱[在所有伺服器中停用 [公告] AllowSynchronousIO （aspnet/AspNetCore #7644）](https://github.com/aspnet/AspNetCore/issues/7644)。

### <a name="microsoftaspnetcoreserverkestrelhttps-assembly-removed"></a>已移除 AspNetCore Kestrel Https 元件

在 ASP.NET Core 2.1 中，AspNetCore 的內容已移至*AspNetCore. Kestrel*. *Kestrel* ... d. d. .dll。 這是使用`TypeForwardedTo`屬性的非重大更新。 若是3.0，則已移除空的*AspNetCore Kestrel*元件（和 NuGet 套件）。」

參考[AspNetCore](https://www.nuget.org/packages/Microsoft.AspNetCore.Server.Kestrel.Https)的程式庫應將 ASP.NET Core 相依性更新為2.1 或更新版本。

以 ASP.NET Core 2.1 或更新版本為目標的應用程式和程式庫，應該移除[Kestrel](https://www.nuget.org/packages/Microsoft.AspNetCore.Server.Kestrel.Https)的任何直接參考。

## <a name="jsonnet-support"></a>Json.NET 支援

在[改善 ASP.NET Core 共用架構](https://blogs.msdn.microsoft.com/webdev/2018/10/29/a-first-look-at-changes-coming-in-asp-net-core-3-0/)的工作中，已從 ASP.NET Core 共用架構中移除[Json.NET](https://www.newtonsoft.com/json/help/html/Introduction.htm) 。

ASP.NET Core 的預設值現在是[system.web](/dotnet/api/system.text.json?view=netcore-3.0)，這是 .net Core 3.0 中的新功能。 可能的話`System.Text.Json` ，請考慮使用。 這是高效能，而且不需要額外的程式庫相依性。 不過，由於`System.Text.Json`是新的，因此目前可能缺少應用程式所需的功能。

如果您的應用`Newtonsoft.Json`程式使用`Newtonsoft.Json`特定的功能，例如 JsonPatch 或轉換器，或其[格式](xref:web-api/advanced/formatting) `Newtonsoft.Json`特定的類型，則可能需要整合。

若要在 ASP.NET Core 3.0 SignalR 專案中使用 Json.NET，請參閱本檔中的[切換為 Newtonsoft。](#switch-to-newtonsoftjson)

若要在 ASP.NET Core 3.0 專案中使用 Json.NET：

* 將套件參考新增至 [Microsoft.AspNetCore.Mvc.NewtonsoftJson](https://nuget.org/packages/Microsoft.AspNetCore.Mvc.NewtonsoftJson)。
* 更新`Startup.ConfigureServices`以呼叫`AddNewtonsoftJson`。

  ```csharp
  services.AddMvc()
      .AddNewtonsoftJson();
  ```

  `AddNewtonsoftJson`與新的 MVC 服務註冊方法相容：

  * `AddRazorPages`
  * `AddControllersWithViews`
  * `AddControllers`

  ```csharp
  services.AddControllers()
      .AddNewtonsoftJson();
  ```

  您可以在呼叫`AddNewtonsoftJson`中設定 Json.NET 設定：

  ```csharp
  services.AddMvc()

             options.SerializerSettings.ContractResolver =
  ```

## <a name="routing-startup-code"></a>路由啟動程式碼

如果應用程式呼叫`UseMvc`或`UseSignalR`，請盡可能將應用程式遷移至[端點路由](xref:fundamentals/routing)。 為了改善與舊版 MVC 的端點路由相容性，我們已還原 ASP.NET Core 2.2 中引進的 URL 產生變更。 如果您在2.2 中使用端點路由時遇到問題，請 ASP.NET Core 3.0 中的改進，但有下列例外狀況：

* 如果應用程式`IRouter`是從`Route`執行或繼承，請使用[DynamicRouteValuesTransformer](https://github.com/aspnet/AspNetCore.Docs/issues/12997)做為取代。

* 如果應用程式直接在`RouteData.Routers` MVC 內部存取以剖析 url，您可以使用來取代此`LinkParser.ParsePathByEndpointName`項。 
 * 使用路由名稱定義路由。
 * 使用`LinkParser.ParsePathByEndpointName`並傳入所需的路由名稱。

端點路由支援與相同的路由模式語法和路由模式撰寫功能`IRouter`。 端點路由支援`IRouteConstraint`。 端點路由支援`[Route]`、 `[HttpGet]`和其他 MVC 路由屬性。

對於大部分的應用程式`Startup` ，只需要變更。

### <a name="migrate-startupconfigure"></a>遷移啟動。設定

建議這麼做：

* 新增`UseRouting`。
* 如果應用程式呼叫`UseStaticFiles`，請`UseStaticFiles` **在之前** `UseRouting`放置。
* 如果應用`AuthorizePage`程式使用或`UseCors` `UseEndpoints` `UseAuthorization` `UseAuthentication` `[Authorize]`之類的驗證/授權功能，請將對和的呼叫放在`UseRouting`和之前：

  ```csharp
  public void Configure(IApplicationBuilder app)
  {
    ...

    app.UseStaticFiles();

    app.UseRouting();
    app.UseCors();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints => {
       endpoints.MapControllers();
    }
  ```

* 將`UseMvc`或`UseSignalR`取代為`UseEndpoints`。
* 如果應用程式使用[cors](xref:security/cors)案例`[EnableCors]`（例如），請`UseCors`將對的呼叫放在使用 cors 的任何其他中介軟體之前（例如`UseAuthentication`， `UseAuthorization`在、 `UseEndpoints`和之前的位置`UseCors` ）。
* 以`IHostingEnvironment` 取代`IWebHostEnvironment` ，並`using` 加入命名空間的語句。<xref:Microsoft.Extensions.Hosting?displayProperty=fullName>
* 取代`IApplicationLifetime`為<xref:Microsoft.Extensions.Hosting.IHostApplicationLifetime> （<xref:Microsoft.Extensions.Hosting?displayProperty=fullName> namespace）。
* 取代`EnvironmentName`為<xref:Microsoft.Extensions.Hosting.Environments> （<xref:Microsoft.Extensions.Hosting?displayProperty=fullName> namespace）。

以下程式碼是典型 ASP.NET Core 2.2 `Startup.Configure`應用程式中的範例：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseStaticFiles();

    app.UseAuthentication();

    app.UseSignalR(hubs =>
    {
        hubs.MapHub<ChatHub>("/chat");
    });

    app.UseMvc(routes =>
    {
        routes.MapRoute("default", "{controller=Home}/{action=Index}/{id?}");
    });
}
```

更新先前`Startup.Configure`的程式碼之後：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseStaticFiles();

    app.UseRouting();

    app.UseCors();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapHub<ChatHub>("/chat");
        endpoints.MapControllerRoute("default", "{controller=Home}/{action=Index}/{id?}");

```

> [!WARNING]
> 對於大部分的應用程式而言`UseAuthentication`， `UseAuthorization`對、 `UseCors`和的呼叫必須出現在`UseRouting`對`UseEndpoints`的呼叫之間，才會生效。

### <a name="security-middleware-guidance"></a>安全性中介軟體指引

對授權和 CORS 的支援與[中介軟體](xref:fundamentals/middleware/index)方法一致。 這可讓您在這些案例中使用相同的中介軟體和功能。 此版本提供了更新的授權中介軟體，並增強了 CORS 中介軟體，讓它能夠瞭解 MVC 控制器所使用的屬性。

#### <a name="cors"></a>CORS

先前，CORS 可能很容易設定。 中介軟體是為了在某些使用案例中使用而提供的，但在其他使用案例中，**不需要**中介軟體就能使用 MVC 篩選器。 在 ASP.NET Core 3.0 中，我們建議所有需要 CORS 的應用程式都使用 CORS 中介軟體搭配端點路由。 `UseCors`可以提供預設原則， `[EnableCors]`而和`[DisableCors]`屬性可用來覆寫所需的預設原則。

在以下範例中：

* 具有`default`已命名原則的所有端點都會啟用 CORS。
* `MyController`類別會`[DisableCors]`使用屬性停用 CORS。

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseCors("default");

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

[DisableCors]
public class MyController : ControllerBase
{
    ...
}
```

#### <a name="authorization"></a>Authorization

在舊版的 ASP.NET Core 中，會透過`[Authorize]`屬性提供授權支援。 授權中介軟體無法使用。 在 ASP.NET Core 3.0 中，需要授權中介軟體。 建議您在之後`UseAuthorization` `UseAuthentication`立即放置 ASP.NET Core 授權中介軟體（）。 您也可以使用可覆寫的預設原則來設定授權中介軟體。

在 ASP.NET Core 3.0 或更新版本`UseAuthorization`中，會`Startup.Configure`在中呼叫， `HomeController`而下列需要已登入的使用者：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

public class HomeController : ControllerBase
{
    [Authorize]
    public IActionResult BuyWidgets()
    {
        ...
    }
}
```

如果應用程式使用`AuthorizeFilter`做為 MVC 中的全域篩選，我們建議在對的`AddAuthorization`呼叫中重構程式碼以提供原則。

最初`DefaultPolicy`設定為需要驗證，因此不需要進行任何額外的設定。 在下列範例中，MVC 端點會標示為`RequireAuthorization` ，以便所有要求都必須以為基礎`DefaultPolicy`來授權。 不過，不`HomeController`需要使用者登入應用程式，即可允許存取， `[AllowAnonymous]`原因如下：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute().RequireAuthorization();
    });
}

[AllowAnonymous]
public class HomeController : ControllerBase
{
    ...
}
```

也可以自訂原則。 以先前的範例為基礎， `DefaultPolicy`設定為需要驗證和特定範圍：

```csharp
public void ConfigureServices(IServiceCollection services)
{
    ...

    services.AddAuthorization(options =>
    {
        options.DefaultPolicy = new AuthorizationPolicyBuilder()
          .RequireAuthenticatedUser()
          .RequireScope("MyScope")
          .Build();
    });
}

public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute().RequireAuthorization();
    });
}

[AllowAnonymous]
public class HomeController : ControllerBase
{
    ...
}
```

或者，所有端點都可以設定為要求授權， `[Authorize]`而`RequireAuthorization`不需要或`FallbackPolicy`藉由設定。 `FallbackPolicy`與不同`DefaultPolicy`。 會由`[Authorize]` 或`RequireAuthorization`觸發，而`FallbackPolicy`不會在設定其他原則時觸發。 `DefaultPolicy` `FallbackPolicy`最初設定為允許未經授權的要求。

下列範例與上述`DefaultPolicy`範例相同，但會`FallbackPolicy`使用來一律要求所有端點的驗證，但指定時`[AllowAnonymous]`除外：

```csharp
public void ConfigureServices(IServiceCollection services)
{
    ...

    services.AddAuthorization(options =>
    {
        options.FallbackPolicy = new AuthorizationPolicyBuilder()
          .RequireAuthenticatedUser()
          .RequireScope("MyScope")
          .Build();
    });
}

public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

[AllowAnonymous]
public class HomeController : ControllerBase
{
    ...
}
```

中介軟體的授權可在沒有架構具有任何特定授權知識的情況下運作。 例如，[健康狀態檢查](xref:host-and-deploy/health-checks)沒有特定的授權知識，但健康情況檢查可以有中介軟體所套用的可設定授權原則。

此外，每個端點都可以自訂其授權需求。 在下列範例中， `UseAuthorization`會`DefaultPolicy`使用來處理授權，但`/healthz`健康情況檢查端點需要`admin`使用者：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints
            .MapHealthChecks("/healthz")
            .RequireAuthorization(new AuthorizeAttribute(){ Roles = "admin", });
    });
}
```

保護會在某些案例中執行。 `UseEndpoint`如果因為遺漏中介軟體而略過授權或 CORS 原則，中介軟體會擲回例外狀況。 分析器支援以提供有關設定錯誤的其他意見反應。

SignalR 中樞的對應現在會在內`UseEndpoints`進行。

將每個中樞`MapHub`對應到。 如同先前的版本，每個中樞都會明確列出。

在下列範例中，會新增`ChatHub` SignalR 中樞的支援：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapHub<ChatHub>();
    });
}
```

有一個新選項可控制用戶端的訊息大小限制。 例如，在中`Startup.ConfigureServices`：

```csharp
services.AddSignalR(hubOptions =>
{
    hubOptions.MaximumReceiveMessageSize = 32768;
});
```

在 ASP.NET Core 2.2 中，您可以設定`TransportMaxBufferSize`和，以有效地控制訊息大小上限。 在 ASP.NET Core 3.0 中，該選項現在只會控制觀察到背壓之前的大小上限。

### <a name="mvc-controllers"></a>MVC 控制器

控制器的對應現在會在內`UseEndpoints`進行。

如果`MapControllers`應用程式使用屬性路由，則新增。 由於路由包含 ASP.NET Core 3.0 或更新版本中許多架構的支援，因此新增屬性路由控制器是加入宣告的。

取代下列各項：

* `MapRoute`熟悉`MapControllerRoute`
* `MapAreaRoute`熟悉`MapAreaControllerRoute`

由於路由現在包含不僅僅是 MVC 的支援，因此術語已變更，讓這些方法清楚地陳述其用途。 傳統路由（例如`MapControllerRoute` ） / `MapAreaControllerRoute` 會依其加入的順序套用/。 `MapDefaultControllerRoute` 先放置更特定的路由（例如某個區域的路由）。

在以下範例中：

* `MapControllers`新增屬性路由控制器的支援。
* `MapAreaControllerRoute`為區域中的控制器新增傳統路由。
* `MapControllerRoute`新增控制器的傳統路由。

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapControllers();
        endpoints.MapAreaControllerRoute(
            "admin",
            "admin",
            "Admin/{controller=Home}/{action=Index}/{id?}");
        endpoints.MapControllerRoute(
            "default", "{controller=Home}/{action=Index}/{id?}");
    });
}
```

## <a name="migrating-libraries-via-multi-targeting"></a>透過多目標的方式來遷移程式庫

程式庫通常需要支援多個版本的 ASP.NET Core。 大部分針對舊版 ASP.NET Core 編譯的程式庫應該會繼續運作，而不會發生問題。 下列條件需要跨編譯應用程式：

* 程式庫依賴具有二進位[中斷性變更](#breaking-api-changes)的功能。
* 程式庫想要利用 ASP.NET Core 3.0 中的新功能。 

例如:

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFrameworks>netcoreapp3.0;netstandard2.0</TargetFrameworks>
  </PropertyGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netcoreapp3.0'">
    <FrameworkReference Include="Microsoft.AspNetCore.App" />
  </ItemGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netstandard2.0'">
    <PackageReference Include="Microsoft.AspNetCore" Version="2.1.0" />
  </ItemGroup>
</Project>
```

使用`#ifdefs`來啟用 ASP.NET Core 3.0 特定 api：

```csharp
var webRootFileProvider =
#if NETCOREAPP3_0
    GetRequiredService<IWebHostEnvironment>().WebRootFileProvider;
#else if NETSTANDARD2_0
    GetRequiredService<IHostingEnvironment>().WebRootFileProvider;
#else
#error unknown target framework
#endif
```
