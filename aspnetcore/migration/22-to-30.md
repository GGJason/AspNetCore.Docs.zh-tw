# <a name="migrate-from-aspnet-core-22-to-30"></a>從 ASP.NET Core 2.2 遷移至3。0

由[Scott Addie](https://github.com/scottaddie)和[Rick Anderson](https://twitter.com/RickAndMSFT)

本文說明如何將現有的 ASP.NET Core 2.2 專案更新為 ASP.NET Core 3.0。 建立新的 ASP.NET Core 3.0 專案可能會很有説明：

* 與 ASP.NET Core 2.2 程式碼比較。
* 將相關的變更複製到您的 ASP.NET Core 3.0 專案。

### <a name="framework-reference"></a>架構參考

如以上所列的其中一個封裝所提供的 ASP.NET Core 功能，可做為 `Microsoft.AspNetCore.App` 共用架構的一部分。 *共用架構*是一組安裝在電腦上的元件（*.dll*檔案），其中包含執行時間元件和目標套件。 如需詳細資訊，請參閱[共用的架構](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/) \(英文\)。

* 以 SDK 為目標的專案會 `Microsoft.NET.Sdk.Web` 隱含地參考 `Microsoft.AspNetCore.App` 架構。

  這些專案不需要其他參考：

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Web">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>
      ...
  </Project>
  ```

* `Microsoft.NET.Sdk`以或 SDK 為目標的專案 `Microsoft.NET.Sdk.Razor` 應該將明確新增 `FrameworkReference` 至 `Microsoft.AspNetCore.App` ：

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Razor">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>

    <ItemGroup>
      <FrameworkReference Include="Microsoft.AspNetCore.App" />
    </ItemGroup>
      ...
  </Project>
  ```

#### <a name="framework-dependent-builds-using-docker"></a>使用 Docker 的 Framework 相依組建

使用相依于 ASP.NET Core[共用架構](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/)之套件的主控台應用程式架構相依組建可能會提供下列執行階段錯誤：

```console
It was not possible to find any compatible framework version
The specified framework 'Microsoft.AspNetCore.App', version '3.0.0' was not found.
  - No frameworks were found.
```

`Microsoft.AspNetCore.App`是包含 ASP.NET Core 執行時間的共用架構，而且只會出現在[dotnet/Core/aspnet](https://hub.docker.com/_/microsoft-dotnet-core-aspnet/) Docker 映射上。 3.0 SDK 會使用 ASP.NET Core，藉由不包含共用架構中可用的程式庫重複複製，來減少 framework 相依組建的大小。 這可能節省高達 18 MB，但需要 ASP.NET Core 執行時間顯示/安裝，才能執行應用程式。

若要判斷應用程式是否具有 ASP.NET Core 共用架構的相依性（直接或間接），請檢查應用程式的組建/發行期間所產生的檔案*runtimeconfig.js* 。 下列 JSON 檔案顯示 ASP.NET Core 共用架構的相依性：

```json
{
  "runtimeOptions": {
    "tfm": "netcoreapp3.0",
    "framework": {
      "name": "Microsoft.AspNetCore.App",
      "version": "3.0.0"
    },
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

如果您的應用程式使用 Docker，請使用包含 ASP.NET Core 3.0 的基底映射。 例如： `docker pull mcr.microsoft.com/dotnet/core/aspnet:3.0` 。

## <a name="kestrel"></a>Kestrel

### <a name="configuration"></a>組態

將 Kestrel 設定遷移至所提供的[web 主機](#hostb)產生器 `ConfigureWebHostDefaults` （*Program.cs*）：

```csharp
public static IHostBuilder CreateHostBuilder(string[] args) =>
    Host.CreateDefaultBuilder(args)
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.ConfigureKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseStartup<Startup>();
        });
```

如果應用程式以手動方式建立主機 `HostBuilder` ，請 `UseKestrel` 在的 web 主機 builder 上呼叫 `ConfigureWebHostDefaults` ：

```csharp
public static void Main(string[] args)
{
    var host = new HostBuilder()
        .UseContentRoot(Directory.GetCurrentDirectory())
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.UseKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseIISIntegration()
            .UseStartup<Startup>();
        })
        .Build();

    host.Run();
}
```

### <a name="connection-middleware-replaces-connection-adapters"></a>連線中介軟體會取代連接介面卡

已從 Kestrel 移除連接介面卡（ `Microsoft.AspNetCore.Server.Kestrel.Core.Adapter.Internal.IConnectionAdapter` ）。 以連線中介軟體取代連接介面卡。 連接中介軟體類似于 ASP.NET Core 管線中的 HTTP 中介軟體，但適用于較低層級的連接。 HTTPS 和連線記錄：

* 已從連接介面卡移至連線中介軟體。
* 這些擴充方法的作用如同舊版的 ASP.NET Core。 

如需詳細資訊，請參閱[Kestrel 文章的 listenoptions 來（通訊協定）一節中的 TlsFilterConnectionHandler 範例](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#listenoptionsprotocols)。

### <a name="kestrel-request-trailer-headers"></a>Kestrel 要求尾端標頭

針對以舊版 ASP.NET Core 為目標的應用程式：

* Kestrel 會將 HTTP/1.1 區塊尾端標頭新增至要求標頭集合。
* 結尾會在讀取要求主體之後提供。

這會導致一些關於標頭和結尾的多義性問題，因此結尾已移至3.0 中的新集合（ `RequestTrailerExtensions` ）。

HTTP/2 要求尾端為：

* 不適用於 ASP.NET Core 2.2。
* 以3.0 提供 `RequestTrailerExtensions` 。

有新的要求擴充方法可存取這些結尾。 如同 HTTP/1.1，結尾會在讀取要求主體之後提供。

針對3.0 版本， `RequestTrailerExtensions` 可以使用下列方法：

* `GetDeclaredTrailers`：取得要求 `Trailer` 標頭，其中列出主體之後預期的尾端。
* `SupportsTrailers`：指出要求是否支援接收尾端標頭。
* `CheckTrailersAvailable`：檢查要求是否支援尾端，以及是否可供讀取。 這種檢查並不會假設有尾端可供讀取。 即使這個方法傳回，也可能沒有尾端可讀取 `true` 。
* `GetTrailer`：從回應中取得要求的尾端標頭。 `SupportsTrailers`請在呼叫之前檢查 `GetTrailer` ， <xref:System.NotSupportedException> 如果要求不支援尾端標頭，則可能會發生。

如需詳細資訊，請參閱[將要求尾端放在個別的集合中（dotnet/AspNetCore #10410）](https://github.com/dotnet/AspNetCore/pull/10410)。

### <a name="allowsynchronousio-disabled"></a>AllowSynchronousIO 已停用

`AllowSynchronousIO`啟用或停用同步 i/o Api，例如 `HttpRequest.Body.Read` 、 `HttpResponse.Body.Write` 和 `Stream.Flush` 。 這些 Api 是執行緒耗盡的來源，導致應用程式損毀。 在3.0 中， `AllowSynchronousIO` 預設為停用。 如需詳細資訊，請參閱[Kestrel 文章中的同步 i/o 一節](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#synchronous-io)。

如果需要同步 i/o，可以 `AllowSynchronousIO` 在使用的伺服器上設定選項（ `ConfigureKestrel` 例如，如果使用 Kestrel，則會啟用）。 請注意，伺服器（Kestrel、HttpSys、TestServer 等）都有自己 `AllowSynchronousIO` 的選項，而不會影響其他伺服器。 針對每個要求，您可以使用選項來啟用所有伺服器的同步 i/o `IHttpBodyControlFeature.AllowSynchronousIO` ：

```csharp
var syncIOFeature = HttpContext.Features.Get<IHttpBodyControlFeature>();

if (syncIOFeature != null)
{
    syncIOFeature.AllowSynchronousIO = true;
}
```

如果您對於 <xref:System.IO.TextWriter> 在[Dispose](/dotnet/standard/garbage-collection/implementing-dispose)中呼叫同步 api 的程式或其他資料流程遇到問題，請改為呼叫新的 <xref:System.IO.Stream.DisposeAsync*> api。

如需詳細資訊，請參閱[在所有伺服器中停用 [公告] AllowSynchronousIO （dotnet/AspNetCore #7644）](https://github.com/dotnet/AspNetCore/issues/7644)。

### <a name="output-formatter-buffering"></a>輸出格式器緩衝

以、和為基礎的輸出格式器[Newtonsoft.Js](https://www.newtonsoft.com/json) <xref:System.Xml.Serialization.XmlSerializer> <xref:System.Runtime.Serialization.DataContractSerializer> 只支援同步序列化。 為了讓這些格式器能夠使用伺服器的[AllowSynchronousIO](https://github.com/dotnet/aspnetcore/issues/7644)限制，MVC 會在寫入磁片之前緩衝處理這些格式器的輸出。 由於緩衝處理的結果，MVC 會在使用這些格式器回應時包含 Content-length 標頭。

<xref:System.Text.Json>支援非同步序列化，因此 `System.Text.Json` 根據格式器不會緩衝。 請考慮使用此格式器來改善效能。

若要停用緩衝處理，應用程式可以 <xref:Microsoft.AspNetCore.Mvc.MvcOptions.SuppressOutputFormatterBuffering> 在其啟動時設定：

```csharp
services.AddControllers(options => options.SuppressOutputFormatterBuffering = true)
```

請注意，如果未同時設定，這可能會導致應用程式擲回執行時間例外狀況 `AllowSynchronousIO` 。

### <a name="microsoftaspnetcoreserverkestrelhttps-assembly-removed"></a>已移除 AspNetCore Kestrel Https 元件

在 ASP.NET Core 2.1 中， *Microsoft.AspNetCore.Server.Kestrel.Https.dll*的內容已移至*Microsoft.AspNetCore.Server.Kestrel.Core.dll*。 這是使用屬性的非重大更新 `TypeForwardedTo` 。 針對3.0，已移除空的*Microsoft.AspNetCore.Server.Kestrel.Https.dll*元件和 NuGet 套件。

參考[AspNetCore](https://www.nuget.org/packages/Microsoft.AspNetCore.Server.Kestrel.Https)的程式庫應將 ASP.NET Core 相依性更新為2.1 或更新版本。

以 ASP.NET Core 2.1 或更新版本為目標的應用程式和程式庫，應該移除[Kestrel](https://www.nuget.org/packages/Microsoft.AspNetCore.Server.Kestrel.Https)的任何直接參考。

## <a name="routing-startup-code"></a>路由啟動程式碼

如果應用程式呼叫 `UseMvc` 或 `UseSignalR` ，請盡可能將應用程式遷移至[端點路由](xref:fundamentals/routing)。 為了改善與舊版 MVC 的端點路由相容性，我們已還原 ASP.NET Core 2.2 中引進的 URL 產生變更。 如果您在2.2 中使用端點路由時遇到問題，請 ASP.NET Core 3.0 中的改進，但有下列例外狀況：

* 如果應用程式是 `IRouter` 從執行或繼承 `Route` ，請使用[DynamicRouteValuesTransformer](https://github.com/dotnet/AspNetCore.Docs/issues/12997)做為取代。
* 如果應用程式直接 `RouteData.Routers` 在 MVC 內部存取以剖析 url，您可以使用[LinkParser. ParsePathByEndpointName](xref:Microsoft.AspNetCore.Routing.LinkParserEndpointNameAddressExtensions.ParsePathByEndpointName*)來取代此項。 
  * 使用路由名稱定義路由。
  * 使用 `LinkParser.ParsePathByEndpointName` 並傳入所需的路由名稱。

端點路由支援與相同的路由模式語法和路由模式撰寫功能 `IRouter` 。 端點路由支援 `IRouteConstraint` 。 端點路由支援 `[Route]` 、 `[HttpGet]` 和其他 MVC 路由屬性。

對於大部分的應用程式，只 `Startup` 需要變更。

### <a name="migrate-startupconfigure"></a>遷移 Startup.Configu

一般建議：

* 加入 `UseRouting`。
* 如果應用程式呼叫 `UseStaticFiles` ，請 `UseStaticFiles` **在之前**放置 `UseRouting` 。
* 如果應用程式使用或之類的驗證/授權 `AuthorizePage` 功能 `[Authorize]` ，請將對和的呼叫放在 `UseAuthentication` 和 `UseAuthorization` **after** `UseRouting` `UseCors` 之前 `UseEndpoints` ：

  ```csharp
  public void Configure(IApplicationBuilder app)
  {
    ...

    app.UseStaticFiles();

    app.UseRouting();
    app.UseCors();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints => {
       endpoints.MapControllers();
    });
  ```

* 將 `UseMvc` 或取代 `UseSignalR` 為 `UseEndpoints` 。
* 如果應用程式使用[cors](xref:security/cors)案例（例如 `[EnableCors]` ），請將對的呼叫放在 `UseCors` 使用 cors 的任何其他中介軟體之前（例如， `UseCors` 在、和之前的位置 `UseAuthentication` `UseAuthorization` `UseEndpoints` ）。
* 以取代， `IHostingEnvironment` `IWebHostEnvironment` 並加入 `using` <xref:Microsoft.Extensions.Hosting?displayProperty=fullName> 命名空間的語句。
* 取代 `IApplicationLifetime` 為 <xref:Microsoft.Extensions.Hosting.IHostApplicationLifetime> （ <xref:Microsoft.Extensions.Hosting?displayProperty=fullName> namespace）。
* 取代 `EnvironmentName` 為 <xref:Microsoft.Extensions.Hosting.Environments> （ <xref:Microsoft.Extensions.Hosting?displayProperty=fullName> namespace）。

以下程式碼是 `Startup.Configure` 典型 ASP.NET Core 2.2 應用程式中的範例：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseStaticFiles();

    app.UseAuthentication();

    app.UseSignalR(hubs =>
    {
        hubs.MapHub<ChatHub>("/chat");
    });

    app.UseMvc(routes =>
    {
        routes.MapRoute("default", "{controller=Home}/{action=Index}/{id?}");
    });
}
```

更新先前的程式 `Startup.Configure` 代碼之後：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseStaticFiles();

    app.UseRouting();

    app.UseCors();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapHub<ChatHub>("/chat");
        endpoints.MapControllerRoute("default", "{controller=Home}/{action=Index}/{id?}");
    });
}
```

> [!WARNING]
> 對於大部分的應用程式而言，對 `UseAuthentication` 、 `UseAuthorization` 和 `UseCors` 的呼叫必須出現在對的呼叫之間，才 `UseRouting` `UseEndpoints` 會生效。

### <a name="security-middleware-guidance"></a>安全性中介軟體指引

對授權和 CORS 的支援與[中介軟體](xref:fundamentals/middleware/index)方法一致。 這可讓您在這些案例中使用相同的中介軟體和功能。 此版本提供了更新的授權中介軟體，並增強了 CORS 中介軟體，讓它能夠瞭解 MVC 控制器所使用的屬性。

#### <a name="cors"></a>CORS

先前，CORS 可能很容易設定。 中介軟體是為了在某些使用案例中使用而提供的，但在其他使用案例中，**不需要**中介軟體就能使用 MVC 篩選器。 在 ASP.NET Core 3.0 中，我們建議所有需要 CORS 的應用程式都使用 CORS 中介軟體搭配端點路由。 `UseCors`可以提供預設原則，而 `[EnableCors]` 和 `[DisableCors]` 屬性可用來覆寫所需的預設原則。

在下例中︰

* 具有已命名原則的所有端點都會啟用 CORS `default` 。
* `MyController`類別會使用屬性停用 CORS `[DisableCors]` 。

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseCors("default");

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

[DisableCors]
public class MyController : ControllerBase
{
    ...
}
```

#### <a name="authorization"></a>授權

在舊版的 ASP.NET Core 中，會透過屬性提供授權支援 `[Authorize]` 。 授權中介軟體無法使用。 在 ASP.NET Core 3.0 中，需要授權中介軟體。 建議您在之後立即放置 ASP.NET Core 授權中介軟體（ `UseAuthorization` ） `UseAuthentication` 。 您也可以使用可覆寫的預設原則來設定授權中介軟體。

在 ASP.NET Core 3.0 或更新版本中， `UseAuthorization` 會在中呼叫 `Startup.Configure` ，而下列 `HomeController` 需要已登入的使用者：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

public class HomeController : Controller
{
    [Authorize]
    public IActionResult BuyWidgets()
    {
        ...
    }
}
```

使用端點路由時，建議您不要設定 `<xref:Microsoft.AspNetCore.Mvc.Authorization.AuthorizeFilter>` ，而是依賴授權中介軟體。  如果應用程式使用 `AuthorizeFilter` 做為 MVC 中的全域篩選，我們建議在對的呼叫中重構程式碼以提供原則 `AddAuthorization` 。

`DefaultPolicy`最初設定為需要驗證，因此不需要進行任何額外的設定。 在下列範例中，MVC 端點會標示為， `RequireAuthorization` 以便所有要求都必須以為基礎來授權 `DefaultPolicy` 。 不過， `HomeController` 不需要使用者登入應用程式，即可允許存取，原因 `[AllowAnonymous]` 如下：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute().RequireAuthorization();
    });
}

[AllowAnonymous]
public class HomeController : Controller
{
    ...
}
```

#### <a name="custom-authorization-handlers"></a>自訂授權處理常式

如果應用程式使用自訂[授權處理常式](xref:security/authorization/policies#authorization-handlers)，則端點路由會將不同的資源類型傳遞給處理常式，而不是 MVC。 預期授權處理常式內容資源屬於類型的處理常式 <xref:Microsoft.AspNetCore.Mvc.Filters.AuthorizationFilterContext> （ [MVC 篩選器所提供](xref:security/authorization/policies#access-mvc-request-context-in-handlers)的資源類型）將需要更新，才能處理類型的資源 <xref:Microsoft.AspNetCore.Routing.RouteEndpoint> （由端點路由指定給授權處理常式的資源類型）。

MVC 仍然 `AuthorizationFilterContext` 會使用資源，因此，如果應用程式使用 MVC 授權篩選和端點路由授權，則可能需要處理這兩種類型的資源。

### SignalR

中樞的對應 SignalR 現在會在內進行 `UseEndpoints` 。

將每個中樞對應到 `MapHub` 。 如同先前的版本，每個中樞都會明確列出。

在下列範例中， `ChatHub` SignalR 會新增中樞的支援：

```csharp
public void Configure(IApplicationBuilder app)

    app.UseRouting();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapHub<ChatHub>();
    });
}
```

有一個新選項可控制用戶端的訊息大小限制。 例如，在 `Startup.ConfigureServices` 中：

```csharp

{
    hubOptions.MaximumReceiveMessageSize = 32768;
});
```

在 ASP.NET Core 2.2 中，您可以設定 `TransportMaxBufferSize` 和，以有效地控制訊息大小上限。 在 ASP.NET Core 3.0 中，該選項現在只會控制觀察到背壓之前的大小上限。

### <a name="mvc-controllers"></a>MVC 控制器

控制器的對應現在會在內進行 `UseEndpoints` 。

`MapControllers`如果應用程式使用屬性路由，則新增。 由於路由包含 ASP.NET Core 3.0 或更新版本中許多架構的支援，因此新增屬性路由控制器是加入宣告的。

取代下列項目：

* `MapControllerRoute` 的 `MapRoute`
* `MapAreaControllerRoute` 的 `MapAreaRoute`

由於路由現在包含不僅僅是 MVC 的支援，因此術語已變更，讓這些方法清楚地陳述其用途。 傳統路由（例如） `MapControllerRoute` / `MapAreaControllerRoute` / `MapDefaultControllerRoute` 會依其加入的順序套用。 先放置更特定的路由（例如某個區域的路由）。

在下例中︰

* `MapControllers`新增屬性路由控制器的支援。
* `MapAreaControllerRoute`為區域中的控制器新增傳統路由。
* `MapControllerRoute`新增控制器的傳統路由。

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapControllers();
        endpoints.MapAreaControllerRoute(
            "admin",
            "admin",
            "Admin/{controller=Home}/{action=Index}/{id?}");
        endpoints.MapControllerRoute(
            "default", "{controller=Home}/{action=Index}/{id?}");
    });
}
```

#### <a name="async-suffix-removal-from-controller-action-names"></a>從控制器動作名稱移除非同步尾碼

在 ASP.NET Core 3.0 中，ASP.NET Core MVC 會 `Async` 從控制器動作名稱中移除尾碼。 路由和連結產生都會受到此新預設值的影響。 例如：

```csharp
public class ProductsController : Controller
{
    public async Task<IActionResult> ListAsync()
    {
        var model = await _dbContext.Products.ToListAsync();
        return View(model);
    }
}
```

在 ASP.NET Core 3.0 之前：

* 先前的動作可在*Products/metrics.listasync*路由存取。
* 需要指定尾碼的連結產生 `Async` 。 例如：

    ```cshtml
    <a asp-controller="Products" asp-action="ListAsync">List</a>
    ```

在 ASP.NET Core 3.0：

* 先前的動作可在*產品/清單*路由存取。
* 連結產生不需要指定 `Async` 尾碼。 例如：

    ```cshtml
    <a asp-controller="Products" asp-action="List">List</a>
    ```

這種變更不會影響使用屬性指定的名稱 [`[ActionName]`](/dotnet/api/microsoft.aspnetcore.mvc.actionnameattribute) 。 在中，您可以使用下列程式碼來停用預設行為 `Startup.ConfigureServices` ：

```csharp
services.AddMvc(options =>
    options.SuppressAsyncSuffixInActionNames = false);
```

#### <a name="changes-to-link-generation"></a>連結產生的變更

如檔中與[舊版路由的差異](xref:fundamentals/routing#differences-from-earlier-versions-of-routing)所述，連結產生有一些差異（ `Url.Link` 例如使用和類似的 api）。 其中包括：

* 根據預設，使用端點路由時，不一定會保留產生之 Uri 中的路由參數大小寫。 這個行為可以使用介面來控制 `IOutboundParameterTransformer` 。
* 產生無效路由（控制器/動作或不存在的頁面）的 URI，將會在端點路由下產生空字串，而不會產生不正確 URI。
* 環境值（來自目前內容的路由參數）不會在與端點路由的連結產生中自動使用。 先前，當產生另一個動作（或頁面）的連結時，會從*目前*的路由環境值推斷未指定的路由值。 使用端點路由時，必須在連結產生期間明確指定所有路由參數。

### <a name="razor-pages"></a>Razor頁面

## <a name="opt-in-to-runtime-compilation"></a>加入執行時間編譯

在 ASP.NET Core 3.0 之前，views 的執行時間編譯是架構的隱含功能。 執行時間編譯會補充 views 的組建階段編譯。 它可讓架構在 Razor 修改檔案時編譯視圖和頁面（*. cshtml*檔案），而不需要重建整個應用程式。 這項功能支援在 IDE 中進行快速編輯，以及重新整理瀏覽器以查看變更的案例。

在 ASP.NET Core 3.0 中，執行時間編譯是加入宣告的案例。 「組建時間編譯」是預設啟用之 view 編譯的唯一機制。 執行時間依賴 Visual Studio 或[dotnet-監看](xref:tutorials/dotnet-watch)Visual Studio Code，以在偵測到*cshtml*檔案的變更時重建專案。 在 Visual Studio 中，會在執行的專案（<kbd>Ctrl + F5</kbd>）中變更為 *.cs*、 *. cshtml*或*razor*檔案，但不會進行調試（<kbd>F5</kbd>），會觸發專案的重新編譯。

若要在您的 ASP.NET Core 3.0 專案中啟用執行時間編譯：

1. 請安裝[AspNetCore Razor 。Microsoft.aspnetcore.mvc.razor.runtimecompilation](https://www.nuget.org/packages/Microsoft.AspNetCore.Mvc.Razor.RuntimeCompilation) NuGet 套件。
1. 更新 `Startup.ConfigureServices` 以呼叫 `AddRazorRuntimeCompilation` ：

    針對 ASP.NET Core MVC，請使用下列程式碼：

    ```csharp
    services.AddControllersWithViews()
        .AddRazorRuntimeCompilation(...);
    ```
 
    針對 ASP.NET Core Razor 頁面，請使用下列程式碼：
 
    ```csharp
    services.AddRazorPages()
        .AddRazorRuntimeCompilation(...);
    ```
 
中的範例 https://github.com/aspnet/samples/tree/master/samples/aspnetcore/mvc/runtimecompilation 顯示在開發環境中有條件地啟用執行時間編譯的範例。

如需檔案編譯的詳細資訊 Razor ，請參閱 <xref:mvc/views/view-compilation> 。

## <a name="migrate-libraries-via-multi-targeting"></a>透過多目標來遷移程式庫

程式庫通常需要支援多個版本的 ASP.NET Core。 大部分針對舊版 ASP.NET Core 編譯的程式庫應該會繼續運作，而不會發生問題。 下列條件需要跨編譯應用程式：

* 程式庫依賴具有二進位[中斷性變更](#breaking-api-changes)的功能。
* 程式庫想要利用 ASP.NET Core 3.0 中的新功能。 

例如：

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFrameworks>netcoreapp3.0;netstandard2.0</TargetFrameworks>
  </PropertyGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netcoreapp3.0'">
    <FrameworkReference Include="Microsoft.AspNetCore.App" />
  </ItemGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netstandard2.0'">
    <PackageReference Include="Microsoft.AspNetCore" Version="2.1.0" />
  </ItemGroup>
</Project>
```

使用 `#ifdefs` 來啟用 ASP.NET Core 3.0 特定 api：

```csharp
var webRootFileProvider =
#if NETCOREAPP3_0
    GetRequiredService<IWebHostEnvironment>().WebRootFileProvider;
#elif NETSTANDARD2_0
    GetRequiredService<IHostingEnvironment>().WebRootFileProvider;
#else
#error unknown target framework
#endif
```

如需在類別庫中使用 ASP.NET Core Api 的詳細資訊，請參閱 <xref:fundamentals/target-aspnetcore> 。
